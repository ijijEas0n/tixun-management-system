## 一、这个系统存了哪些数据？

整个系统的数据可以理解为4大本"账本"：

### 📕 账本1：学年花名册
记录有哪些学年（比如"2025年"、"2026年"），每个学年只有一个名字和一个编号。

### 📗 账本2：学生名册
每个学生有：
- **学号**：自动生成的，比如"25001"表示2025年第1号学生
- **姓名**
- **性别**：男/女（非常重要！男女评分标准不同）
- **所属学年**

### 📘 账本3：成绩记录簿
按学生归类，每个学生可以有多条测试记录，每条记录包含：
- **什么时候测的**（日期）
- **属于哪次测试会话**（可选）
- **4个项目各自的成绩**：
  - 100米跑了多少秒
  - 铅球扔了多少米
  - 三级跳跳了多少米
  - 800米跑了多少秒
- **4个项目各自的尝试记录**：比如100米跑了3次，每次多少秒
- **4个项目各自的得分**：根据评分表换算的积分（每项0-25分）
- **总分**：4项积分之和（满分100）

### 📙 账本4：测试会话安排表
每次正式测试的安排，包含：
- **会话名称和日期**
- **谁缺席了**
- **4个项目各自的分组情况**：
  - 可以有多个"版本"（比如重新分了一次组）
  - 当前显示哪个版本
  - 正在用哪个版本录入成绩（录入后分组就"锁定"了）
  - 每个版本里有若干"组"，每组有若干学生
  - 100米每组还有"道次"（第1道、第2道...）
- **每个项目允许试几次**：100米默认3次，800米默认1次
- **时间安排**：几点开始，每组间隔几分钟

## 二、系统能做什么操作？（按用户使用顺序）

### 🏫 学年管理

| 用户操作 | 系统做了什么 |
|---------|------------|
| 新建一个学年 | 起个名字（如"2025"），加到学年列表 |
| 删除一个学年 | ⚠️ 会连带着删掉这个学年下所有学生、所有成绩记录、所有测试会话 |
| 改学年名称 | 直接改名字，学号前缀不会自动跟着改 |

### 👤 学生管理

| 用户操作 | 系统做了什么 |
|---------|------------|
| 手动添加一个学生 | 填姓名和性别，系统自动生成学号（年份后两位+3位序号，如25001） |
| Excel批量导入学生 | 逐行读取Excel，如果"学号+姓名"完全一样就跳过（去重），否则新增 |
| 修改学生信息 | 可以改姓名、性别，甚至改所属学年 |
| 删除学生 | 删掉该学生本人 + 他的所有成绩记录 + 从所有分组中移除 + 从缺席名单中移除 |
| 批量删除学生 | 同上，但一次删多个 |

### 📋 测试会话管理

| 用户操作 | 系统做了什么 |
|---------|------------|
| 新建测试会话 | 填名称、日期、选学年、勾缺席学生，创建空的分组 |
| 修改会话 | 可以改名称、日期等任何字段 |
| 删除会话 | 删掉会话本身，⚠️ 但不会删掉学生已有的成绩记录 |

### 🏃 分组操作

| 用户操作 | 系统做了什么 |
|---------|------------|
| 自动分组 | **800米**：所有人混在一起随机分组<br>**其他项目**：先按性别分开，再各自随机分组，男生组排前面 |
| 按人数分组 | 每N人一组，最后一组可能不满 |
| 按组数分组 | 轮流分配到指定数量的组里 |
| 拖拽调整 | 手动调整组内顺序，100米会自动重排道次 |
| 交换学生 | 两个学生互换位置 |
| 删除组 | 把整组去掉 |
| 锁定分组 | 一旦开始录入成绩，分组就锁定了，不能再改 |

### ⏱ 时间安排

| 用户操作 | 系统做了什么 |
|---------|------------|
| 设置开始时间和间隔 | 比如第1组8:00开始，每组间隔15分钟，则第2组8:15、第3组8:30... |
| 超过午夜 | 自动显示"次日 HH:MM" |

### ✍️ 成绩录入

| 用户操作 | 系统做了什么 |
|---------|------------|
| 手动录入 | 选择会话→项目→分组，逐人逐次填入成绩 |
| 语音录入 | 打开麦克风→说中文→系统自动识别→自动填入对应学生的成绩框 |
| 保存成绩 | 系统自动从多次试投中选最佳成绩，换算成积分，写入学生记录 |

**语音识别怎么工作？**
1. 腾讯云语音SDK把说的话转成文字
2. 按逗号/句号把文字分成小段
3. 每段里找学生名字（或学号、道次号）
4. 提取数字作为成绩
5. 识别"第X次"来填到对应试投次

**语音能听懂什么？**
- 中文数字：一二三、十二、二十三、一百零五
- 小数：三点五 = 3.5
- 时间：一分三十秒 = 1:30、2:15.3
- 位置：3道、5号
- 试次：第2次

### 📊 评分怎么算？

评分表是根据官方PDF（《体考四项分值表》）生成的，存在代码里。

**核心规则**：
- 100米/800米：**越快分越高**（11.54秒满分25分，15.00秒0分）
- 铅球/三级跳：**越远分越高**（12.00米满分25分，5.00米低分）
- 男女分别有不同的评分标准
- 每项满分25分，4项总分满分100分
- 没参加的项目得0分

**查表过程**（举例：男生100米跑了11.54秒）：
1. 成绩转换成百分位整数：11.54 → 1154
2. 查表：`points[1154 - 1154]` = `points[0]` = 2500
3. 除以100：2500 → 25.00分

### 🏆 排行与分析

|------|------|

## 三、⚠️ 发现的问题和风险

### 🔴 严重问题（可能导致数据出错或丢失）

| # | 问题 | 大白话解释 | 后果 |
|---|------|-----------|------|
| 1 | ID生成方式不安全 | 系统用"当前时间+随机数"来给每条数据生成唯一编号，但快速操作时时间戳相同，可能撞号 | 两个学生/记录变成同一个ID，后者覆盖前者，数据丢失 |
| 2 | 批量保存成绩ID更易撞 | 同一批次内"当前时间"完全一样，全靠随机数区分 | 同一学生可能多条新记录ID相同 |
| 3 | 预编排导入"替换"模式不可撤销 | 点了替换就把整年数据清空，没有备份、没有确认弹窗、没有撤销 | 一手误，整年学生+成绩+会话全没了 |
| 4 | 导入学生ID不稳定 | 如果学号里有空格或特殊符号，系统编码方式不一致 | 同一个学生两次导入可能被当成两个人 |
| 5 | 删学年可能留孤儿数据 | 删学年会删学生和成绩，但如果其他地方还引用了这些学生，引用就变成"空指针" | 页面显示异常或报错 |
| 6 | 负数成绩可能得正分 | 评分逻辑没有排除负数，比如铅球成绩填-5米，可能还能查表得分 | 明显错误的成绩没被拦截 |
| 7 | 浮点精度导致边界算错 | 计算机处理小数有精度问题，比如11.54在内部可能存成11.539999，转百分位时差1 | 边界成绩可能查错表，差0.01分 |
| 8 | 缺考项目=0分 | 没参加的项目直接0分计入总分 | 缺考学生和全参加但成绩差的学生无法区分 |

### 🟡 中等问题（可能导致数据不一致或体验差）

| # | 问题 | 大白话解释 | 后果 |
|---|------|-----------|------|
| 9 | 快速添加学生学号可能重复 | 添加学生时读的是"旧数据快照"来算学号，不是最新数据 | 快速连点"添加"可能两个学生分到同一个学号 |
| 10 | 删学年后当前年切换可能出错 | 同上，判断"当前选中年"用的是旧数据 | 删除后可能页面空白 |
| 11 | 修改学生学年不迁移数据 | 把学生从2025年改到2026年，成绩记录还留在2025年，分组里也还有他 | 数据错乱：学生2026年但成绩在2025年 |
| 12 | 修改会话学年也类似 | 把测试会话改到另一个年，但分组里的学生还是旧年的 | 同上 |
| 13 | 没有任何输入验证 | 学年可以填空名字，学生可以没名字，日期可以乱填 | 各种显示异常 |
| 14 | 成绩没有范围检查 | 100米可以填0.01秒，铅球可以填999米 | 错误成绩不拦截，直接算分 |
| 15 | 语音识别中等信心结果直接用 | 语音识别的"不太确定"结果也直接填入了 | 说错的成绩被录入 |
| 16 | 导入默认性别为"男" | Excel里没写性别的学生自动当男生 | 女生被当男生评分，分数偏低（男生评分标准更严） |
| 17 | 数据量大时变慢 | 成绩分析每次打开都全量重算，不缓存 | 学生/成绩多了排行页卡顿 |
| 18 | 启动时全量重算积分 | 每次打开App都把所有记录重算一遍 | 数据多了启动慢 |

### 🟢 低风险问题

| # | 问题 | 说明 |
|---|------|------|
| 19 | 腾讯云密钥明文存在浏览器 | 任何能打开浏览器的人都能看到API密钥 |
| 20 | 所有数据明文无加密 | 浏览器里可以看到和修改所有数据 |
| 21 | 800米时间可以输入不合理的值 | 比如999分99秒也能通过 |
| 22 | 导入时文件名没日期就默认今天 | 用户可能不知道日期被自动推断 |
| 23 | 进步/退步分析可能比较了不相关的测试 | 系统找"上一次测试"的逻辑可能跨会话 |

## 四、数据怎么存？存在哪？

**全部存在浏览器的 localStorage 里**，没有服务器、没有数据库。

- 优点：不需要网络、打开浏览器就能用
- 缺点：
  - 数据只在这台电脑这个浏览器里，换台电脑就没有了
  - 浏览器清缓存就全没了
  - 容量大约5MB，学生多了可能不够
  - 任何JS脚本都能读写，不安全

## 一、数据分类体系

### 1.1 根数据结构 (AppData)
```
AppData {
  years: AcademicYear[]        // 学年列表
  students: Student[]          // 学生列表
  records: Record<string, TestRecord[]>  // 学生ID → 测试记录数组
  testSessions: TestSession[]  // 测试会话列表
}
```

### 1.2 各数据实体

| 实体 | 主键 | 核心字段 | 关联关系 |
|------|------|---------|---------|
| AcademicYear | id | name | ← students.yearId, ← testSessions.yearId |
| Student | id | studentNo, name, gender, yearId | → yearId, → records[studentId] |
| TestRecord | id | date, testSessionId?, testName?, scores, points | → studentId(key), → testSessionId? |
| TestSession | id | name, date, yearId, absentStudentIds, groupingVersions, trialConfigs | → yearId, → absentStudentIds[] |

### 1.3 嵌套数据结构

```
TestSession
  ├── activeVersionIds: Partial<Record<SportEventKey, string>>   // 每个项目的当前版本ID
  ├── entryVersionIds: Partial<Record<SportEventKey, string>>    // 每个项目的录入版本ID（锁定标记）
  ├── groupingVersions: Record<SportEventKey, TestSessionGroupingVersion[]>
  │     └── TestSessionGroupingVersion
  │           ├── id, name, event, createdAt, source, mode
  │           ├── groupSize? / groupCount?
  │           └── groups: TestSessionGroup[]
  │                 ├── id, name, marker?, startTime?, gender
  │                 └── members: TestSessionGroupMember[]
  │                       ├── studentId, lane?, order?, rank?, note?
  ├── absentStudentIds: string[]
  ├── trialConfigs: Record<SportEventKey, number>               // 每个项目的试投/试跳次数
  └── groupScheduleConfigs: Partial<Record<SportEventKey, GroupScheduleConfig>>
        ├── startTime: string
        └── intervalMinutes: number
```

```
TestRecord
  ├── id, date, testSessionId?, testName?
  ├── scores: ScoreSet
  │     ├── hundred / shotPut / tripleJump / eightHundred   // 最佳成绩
  │     └── hundredAttempts? / shotPutAttempts? / tripleJumpAttempts? / eightHundredAttempts?  // 每次尝试
  └── points: ScorePoints
        ├── hundred / shotPut / tripleJump / eightHundred   // 各项积分
        └── total                                          // 总积分
```

### 1.4 评分表数据

```
SCORE_TABLES: {
  male: { hundred, shotPut, tripleJump, eightHundred },
  female: { hundred, shotPut, tripleJump, eightHundred }
}
每个 ScoreTable:
  ├── betterIsHigher: boolean   // 铅球/三级跳=true, 100米/800米=false
  ├── min: number               // 最小百分位值(如男100米=1154即11.54秒)
  ├── max: number               // 最大百分位值(如男100米=1500即15.00秒)
  └── points: number[]          // 每个百分位对应的分数(百分位存储，2500=25.00分)
```

## 二、全量操作功能逻辑

### 2.1 学年管理 (storage.ts)

| 操作 | 函数 | 逻辑 |
|------|------|------|
| 新增学年 | `addYear(name)` | `id=Date.now()`, 追加到 years 数组 |
| 删除学年 | `deleteYear(id)` | 1) 若当前年=被删年则切换 2) 找出该年所有学生ID 3) 删除这些学生的records 4) 删除这些学生 5) 删除该年testSessions |
| 重命名学年 | `updateYear(id, name)` | 直接替换 name 字段 |

### 2.2 学生管理 (storage.ts)

| 操作 | 函数 | 逻辑 |
|------|------|------|
| 添加学生 | `addStudent(name, gender, yearId)` | 取年份后两位做前缀，找该年最大学号+1，生成 studentNo，`id=Date.now()+Math.random()` |
| 批量添加学生 | `batchAddStudents(list, yearId)` | 在 setData 回调内：1) 收集已用学号集合 2) 去重(学号+姓名组合 或 纯姓名) 3) 自增生成新学号 4) 追加新学生 |
| 更新学生 | `updateStudent(id, updates)` | 直接 merge 更新字段 |
| 删除学生 | `deleteStudent(id)` | 调用 batchDeleteStudents |
| 批量删除学生 | `batchDeleteStudents(ids)` | 1) 删除这些学生的records 2) 从students过滤掉 3) 从testSessions的absentStudentIds中移除 4) 从所有分组版本的members中移除 |

### 2.3 测试记录管理 (storage.ts)

| 操作 | 函数 | 逻辑 |
|------|------|------|
| 批量更新记录 | `updateRecordsBatch(updates)` | 遍历每个update：1) 找学生 2) 在该学生records中查找匹配记录(按testSessionId或date) 3) 若找到则合并scores并重新算分 4) 若没找到则新建记录 |
| 删除记录 | `deleteRecord(studentId, recordId)` | 从该学生的records数组中过滤掉 |
| 撤销成绩同步 | `revertScoreSyncBatch(target, event, snapshots)` | 调用 scoreSync 的 applyScoreSyncUndoSnapshots |

### 2.4 测试会话管理 (storage.ts)

| 操作 | 函数 | 逻辑 |
|------|------|------|
| 新增会话 | `addTestSession(name, date, yearId, absentIds)` | 创建空分组版本 + 默认试投次数配置 |
| 更新会话 | `updateTestSession(id, updates)` | 直接 merge 更新字段 |
| 删除会话 | `deleteTestSession(id)` | 直接从数组中过滤掉 |
| 添加分组版本 | `addGroupingVersion(sessionId, event, version)` | 追加到该项目的版本列表，同时设为 activeVersion |
| 更新分组版本 | `updateGroupingVersion(sessionId, event, versionId, updates)` | 直接 merge 更新字段 |

### 2.5 预编排导入 (prearrangedImport.ts + storage.ts)

| 操作 | 函数 | 逻辑 |
|------|------|------|
| 解析Excel | `parsePrearrangedWorkbook(sheets, options)` | 1) 从非项目sheet提取学生(姓名+编号) 2) 从项目sheet解析分组 3) 自动推断日期(从文件名) 4) 构建TestSession和分组版本 |
| 应用导入 | `applyPrearrangedImport(yearId, result, mode)` | 调用 mergePrearrangedImportWithYearData |
| 替换模式 | `replaceYearDataWithPrearrangedImport` | 删除该年所有学生/记录/会话，替换为导入数据 |
| 追加/仅链接 | `mergePrearrangedImportWithYearData` | 按学号+姓名匹配已有学生，匹配到的重写ID映射，未匹配的追加 |

### 2.6 评分计算 (scoring.ts + scoringTables.ts)

| 操作 | 函数 | 逻辑 |
|------|------|------|
| 计算全部分数 | `calculatePoints(scores, gender)` | 查4个评分表分别计算，求和 |
| 计算单项分 | `calculateSingleEventPoint(value, gender, event)` | 查单项评分表 |
| 查表算分 | `pointFromTable(value, table)` | 1) null/≤0/NaN → 0 2) 百分位转换 3) 判断是否越界(满分25或0) 4) `points[normalizedValue - min]` 查表 5) 除以100保留2位小数 |

**评分表结构**：值存储为百分位整数（11.54→1154），分值也存为百分位（25.00→2500），每个数组元素对应0.01的步进。

### 2.7 分组逻辑 (grouping.ts)

| 操作 | 函数 | 逻辑 |
|------|------|------|
| 生成分组版本 | `createGroupingVersion(opts)` | 800米：全体混合随机分组；其他项目：男女分别洗牌分组 |
| 按大小分 | `chunkBySize(students, size)` | 每size人一组 |
| 按组数分 | `splitByCount(students, count)` | 轮流分配到count组 |

## 三、🚨 发现的漏洞与风险

### 🔴 严重 — 数据损坏/丢失风险

**1. ID 生成使用 `Date.now() + Math.random()` 不安全**
- 位置：`storage.ts` L124, L249, L300, L415, L449
- 问题：快速连续操作时 `Date.now()` 可能相同，`Math.random()` 碰撞虽概率低但无保证
- 风险：两个学生/记录/会话可能生成相同 ID，导致数据覆盖
- 建议：使用 `crypto.randomUUID()` 或添加递增计数器

**2. 删除学年后删除学生记录，但可能残留孤儿数据**
- 位置：`storage.ts` L205-227
- 问题：`deleteYear` 删除学生和他们的 records，但 testSessions 中如果分组版本的 members 引用了这些学生，那些引用被清除了（L327-341），但如果有其他地方引用则可能残留
- 风险：孤儿记录引用

**3. `updateRecordsBatch` 新记录的 ID 碰撞**
- 位置：`storage.ts` L415
- 问题：`id: Date.now().toString() + Math.random()` 在批量处理中循环调用，同一批内的 Date.now() 完全相同，Math.random() 概率碰撞
- 风险：同一学生的多条新记录可能 ID 相同，后者覆盖前者

**4. 预编排导入替换模式不可撤销**
- 位置：`prearrangedImport.ts` L355-378
- 问题：`replaceYearDataWithPrearrangedImport` 直接删除该年所有学生/记录/会话，无备份无确认
- 风险：一键误操作可清空整年数据

**5. 预编排导入学生 ID 格式不稳定**
- 位置：`prearrangedImport.ts` L151-154
- 问题：`makeStudentId` 使用 `encodeURIComponent(key)`，如果学号含特殊字符（空格、+等）可能产生不一致的 ID
- 风险：同一学生可能在不同导入中生成不同 ID

### 🔴 严重 — 计分逻辑漏洞

**6. `pointFromTable` 对越界值处理不完善**
- 位置：`scoring.ts` L8-23
- 问题：
  - 对于 `betterIsHigher=true` 的项目（铅球/三级跳），`normalizedValue >= table.max` 得满分25，但 `normalizedValue > table.max` 也得25分
  - 这意味着超出满分成绩线依然得25分，逻辑正确但缺少"超满分"的额外奖励或标记
  - 更严重的是：如果成绩为负数但 `> table.min`（对 betterIsHigher=true），仍会查表得分
- 风险：极端异常值可能得到不合理分数

**7. `toHundredths` 浮点精度问题**
- 位置：`scoring.ts` L4-6
- 问题：`Math.round((value + Number.EPSILON) * 100)` 对某些浮点数可能产生偏移
- 例：理论上 11.54 → 1154，但某些浮点运算后可能得到 1153 或 1155
- 风险：边界成绩可能算错分数（查错表索引）

**8. `calculatePoints` 对 null 成绩返回 0 分而非排除**
- 位置：`scoring.ts` L25-39
- 问题：未参加的项目得 0 分并计入总分，这意味着少参加项目的学生总分必然很低
- 风险：排行时部分项目缺考的学生和全部参加但成绩差的学生无法区分（是否为设计意图存疑）

### 🟡 中等 — 状态一致性问题

**9. `addStudent` 使用 `data` 而非 `prev` 读取当前状态**
- 位置：`storage.ts` L237-258
- 问题：`addStudent` 在 setData 外部读取 `data.years` 和 `data.students` 来计算学号，但在 setData 内部使用的是闭包捕获的旧值
- 风险：快速连续添加学生时，学号可能重复（因为都读到了同一个 data 快照）
- 对比：`batchAddStudents` 正确地在 setData 回调内读取 `prev`

**10. `deleteYear` 使用 `data` 而非 `prev` 判断当前年**
- 位置：`storage.ts` L206
- 问题：`if (currentYearId === id)` 使用外部 `data`，但 `setData` 内用的是 `prev`，存在不一致
- 风险：理论上可能导致删除当前选中年后无法正确切换

**11. 分组版本删除后 activeVersionId/entryVersionId 悬空**
- 位置：`storage.ts` - 缺少
- 问题：没有删除单个分组版本的操作。如果未来添加，需同时清理 activeVersionIds 和 entryVersionIds 中的引用，否则指向不存在的版本
- 风险：当前无此操作，但缺少防护机制

**12. `updateStudent` 可修改 yearId 导致数据不一致**
- 位置：`storage.ts` L312-317
- 问题：`updateStudent` 允许修改 yearId，但不会迁移该学生的 records，也不会更新 testSessions 中的引用
- 风险：学生转年后，旧年记录仍在，但旧年分组中仍有该学生

**13. `updateTestSession` 可修改 yearId 导致数据不一致**
- 位置：`storage.ts` L464-471
- 问题：允许通过 `...updates` 修改 yearId，但该会话的分组中引用的学生可能属于不同年份
- 风险：会话移到新年后，原年学生仍在分组中

### 🟡 中等 — 输入验证缺失

**14. 学年名称无验证**
- 位置：`storage.ts` L200-203
- 问题：`addYear(name)` 不检查空字符串、纯空格、重复名称
- 风险：可创建空名称或重名学年，导致学号前缀异常

**15. 学生姓名无验证**
- 位置：`storage.ts` L236-258
- 问题：`addStudent` 不检查空姓名、特殊字符、超长姓名
- 风险：空姓名学生导致分组显示异常

**16. 测试日期无验证**
- 位置：`storage.ts` L447-462
- 问题：`addTestSession` 接受任意 date 字符串，不验证格式
- 风险：无效日期导致排序和显示异常

**17. 成绩值无范围校验**
- 位置：`storage.ts` L366-428, `scoreSync.ts` L54-57
- 问题：`updateRecordsBatch` 和 `buildScoreSyncUpdates` 不检查成绩是否在合理范围内（如100米0.5秒、铅球100米等极端值）
- 风险：录入错误值不会报警，可能产生0分或满分的不合理结果

**18. 语音解析结果无二次确认**
- 位置：`voiceScoreParser.ts` 整体
- 问题：语音解析的 medium confidence 结果仍被使用，且无阈值过滤
- 风险：误识别的成绩被直接录入

### 🟡 中等 — 性能问题

**19. `batchDeleteStudents` 对 testSessions 的 O(n²) 遍历**
- 位置：`storage.ts` L319-345
- 问题：对每个 testSession 都遍历所有 event 的所有 version 的所有 group 的所有 member 进行过滤
- 风险：大量学生 + 大量会话时性能差

**20. `parseStoredAppData` 每次加载都重算所有记录的 points**
- 位置：`storage.ts` L162-176
- 问题：应用启动时遍历所有学生所有记录重新计算积分
- 风险：数据量大时启动缓慢

**21. `performanceAnalysis.ts` 大量 flatMap 和排序**
- 位置：整个文件
- 问题：`buildOverallPerformanceAnalysis` 等函数每次调用都全量计算
- 风险：数据量大时排行页面卡顿
